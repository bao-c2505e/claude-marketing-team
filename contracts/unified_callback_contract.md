# Unified Callback Contract

**Version:** unified_callback_v0.1

---

## 1. Goal & Objectives
This contract standardizes the callback payload structure dispatched from n8n / specialist modules back to the Core product. It establishes an approval gate contract that governs how module outputs are validated, reviewed, and finalized before triggering downstream actions.

This applies to all integrated modules:
- `creative_asset_comfyui`
- `content_pack_generator`
- `ads_pack_generator`
- `crm_followup_generator`
- `analytics_report_generator`

---

## 2. Callback Payload Standard
All callbacks sent to the Core must conform to the following schema:

### Required Fields
- `contract_version` (string): Must be `"unified_callback_v0.1"`.
- `request_id` (string): Unique UUIDv4 matching the initial request.
- `event_type` (string): The type of work requested (e.g. `creative_asset.requested`).
- `brand_id` (string): The brand ID (strictly `"brand_demo_001"` in mock mode).
- `campaign_id` (string): Campaign ID.
- `module_id` (string): The ID of the executing module.
- `module_status` (string): The execution status of the module.
- `approval_status` (string): The current state of the approval gate review.
- `output` (object): The output payload generated by the module.
- `errors` (array of strings): A list of error messages, empty if successful.
- `metadata` (object): Additional metadata parameters.
- `source` (string): The identifier of the payload source.
- `generated_at` (string): ISO-8601 timestamp.
- `notes` (string): Remarks and disclaimers.

---

## 3. Allowed Field Values

### A. `approval_status`
- `pending_approval`: Task output has been generated but is awaiting review.
- `approved`: Task output has been reviewed and accepted.
- `rejected`: Task output was reviewed and rejected.
- `needs_revision`: Task output requires modifications.

### B. `module_status`
- `mock_completed`: Module completed processing using local mock parameters.
- `completed`: Module completed processing successfully.
- `failed`: Module failed execution.
- `skipped`: Module execution was skipped.

### C. `source` (Allowed / Recommended)
- `n8n_n8_approval_gate_mock`
- `local_mock_stub`
- `n8n_n7_full_multi_module_stub_integration`

---

## 4. Approval Decision Schema
When a reviewer (e.g., owner or system agent) makes an approval gate decision, the decision payload must match the following schema:

- `approval_id` (string): Unique identifier for the review action.
- `request_id` (string): Matches the initial request ID.
- `decision` (string): One of: `"approved"`, `"rejected"`, `"needs_revision"`, `"pending"`.
- `reviewer` (string): Identifier of the reviewer (e.g. `"owner_mock"`).
- `reviewed_at` (string): ISO-8601 timestamp.
- `reason` (string): Rationale for the decision.
- `revision_notes` (string): Notes explaining required changes if `decision` is `"needs_revision"`.
- `safety_flags` (object): Safety check flags (e.g., brand check, copyright check).
- `next_action` (string): Suggested next step based on the decision.

---

## 5. Decision & State Mapping

The decision mapping determines the `final_status` and next steps for the callback workflow:

| decision | approval_status | final_status | next_action |
| :--- | :--- | :--- | :--- |
| `"approved"` | `approved` | `ready_for_mock_callback_preview` | Prepare mock callback preview (no real callback is sent; future integration requires separate explicit integration phase and owner approval) |
| `"rejected"` | `rejected` | `stopped_rejected` | Stop flow, log rejection, do not trigger callback |
| `"needs_revision"`| `needs_revision`| `revision_required` | Return draft to module or queue for human edit |
| `"pending"` | `pending_approval` | `waiting_for_owner_approval` | Hold queue, wait for owner action |

---

## 6. Safety & Security Rules
1.  **No Auto-Post:** Under no circumstances should marketing assets or social copy generated by stubs be published directly to public channels without an explicit review status.
2.  **No Real Ads spend:** Ad generation runs purely in mock budget mode. No real ad campaigns can be created or funded.
3.  **No Real Customer Messaging:** No emails, SMS, or CRM communications can be dispatched to real addresses or phone numbers.
4.  **No Production Webhooks:** Do not fire callbacks to production servers or production API endpoints.
5.  **No Real Customer Data:** Mock events and outputs must only use synthetic mock profiles.
6.  **No Secrets:** Workflows and mock servers must run with zero credentials.
7.  **Owner Approval:** Production rollout and live webhook callbacks require explicit owner sign-off.
8.  **Error Normalization:** Failed module outputs should be normalized into error callback previews using [error_handling_retry_logging_contract.md](file:///c:/Users/DELL/claude-marketing-team/contracts/error_handling_retry_logging_contract.md).
9.  **Manual Review for Errors:** The approval gate should not approve failed/error outputs without manual review.
