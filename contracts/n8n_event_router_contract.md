# n8n Event Router Contract

**Version:** n8n_event_router_v0.1

---

## 1. Goal & Objectives
This contract defines the integration interface between the Core product and the n8n event router (PC2 workstream). It ensures stable, standardized, and safe routing of events emitted by the Core to their respective downstream processing modules without exposing secrets, hardcoding brand-specific runtimes, or invoking unauthorized real-world API side-effects.

---

## 2. Standard Event Envelope (Core → n8n)
All events dispatched from the Core to the n8n router must conform to the following schema structure:

### Required Fields
- `contract_version` (string): Standard version identifier. Must be `"n8n_event_router_v0.1"`.
- `event_type` (string): The identifier of the event (e.g. `creative_asset.requested`).
- `request_id` (string): A unique UUIDv4 string generated by the Core to track the request lifecycle.
- `brand_id` (string): The identifier of the brand. For development/testing, strictly use `"brand_demo_001"`.
- `campaign_id` (string): The identifier of the marketing campaign.
- `requested_by` (string): The identifier/email of the user or system agent that triggered the request.
- `callback_url` (string): The HTTP/mock endpoint where n8n or the module should deliver the result.
- `payload` (object): Module-specific parameters representing the work requested.
- `metadata` (object): Auxiliary key-value pairs (e.g. workflow configurations, negative prompts).

---

## 3. Supported Event Types & Payload Schemas
The router supports the following five event types:

### A. `creative_asset.requested`
Triggers the generation of visual creative assets (e.g., images).
*   **Payload structure:**
    *   `asset_type` (string): Type of asset to generate (e.g., `"image"`).
    *   `creative_brief` (string): Details about the visual prompt.
    *   `dimensions` (string): Resolution (e.g., `"1024x1024"`).
    *   `style_notes` (string): Style guide comments (e.g., `"glassmorphism, vibrant colors"`).

### B. `content_pack.requested`
Triggers text/copy generation for social channels.
*   **Payload structure:**
    *   `channels` (array of strings): Platforms to target (e.g., `["facebook", "instagram"]`).
    *   `number_of_posts` (integer): Total posts to write.
    *   `tone` (string): Personality of copy (e.g., `"professional"`, `"casual"`).
    *   `campaign_theme` (string): Context/topic of the posts.

### C. `ads_pack.requested`
Triggers advertising copy and configuration generation.
*   **Payload structure:**
    *   `platform` (string): Target platform (e.g., `"meta"`, `"google"`).
    *   `objective` (string): Conversion objective (e.g., `"leads"`, `"conversions"`).
    *   `target_audience` (string): Demographics/interest notes.
    *   `offer` (string): The promotional hook or discount deal.
    *   `budget_mode` (string): Control mode, must be `"mock"`.

### D. `crm_followup.requested`
Triggers CRM message sequences or email draft generation.
*   **Payload structure:**
    *   `customer_segment` (string): Target audience segment.
    *   `scenario` (string): Triggers (e.g., `"abandoned_cart"`, `"post_purchase"`).
    *   `message_count` (integer): Sequence length.
    *   `tone` (string): Brand voice styling.
    *   `safety_mode` (string): Control mode, must be `"draft_only"`.

### E. `analytics_report.requested`
Triggers marketing performance report compile task.
*   **Payload structure:**
    *   `data_source` (string): Must be `"mock"`.
    *   `reporting_period` (string): e.g., `"last_30_days"`.
    *   `metrics` (array of strings): Desired KPIs (e.g., `["CTR", "CPC"]`).
    *   `report_format` (string): Output structure (e.g., `"pdf"`, `"markdown"`).

- `approved` → `ready_for_mock_callback_preview` (prepares mock callback preview, no real callback sent)
- `rejected` → `stopped_rejected`
- `needs_revision` → `revision_required`
- `pending_approval` → `waiting_for_owner_approval`

## 6. Safety & Security Rules
1.  **No Secrets:** No real API keys, passwords, or credentials can be embedded in workflows, contracts, or examples.
2.  **No Production URLs:** Do not reference real production endpoints like `thecoreagency.com` in routing tables, JSON files, or mock flows.
3.  **No Real Customer Data:** Always use synthetic names, generic campaigns, and placeholders.
4.  **No Auto-Post/Auto-Send:** Modules should never post copy directly to live channels or send emails to real addresses.
5.  **No Real Ad Spend:** Ad modules must run with `budget_mode: "mock"`.
6.  **Owner Approval:** Production rollout and real execution credentials require explicit owner authorization.
7.  **Unsupported Events Routing:** Unsupported `event_type` routes to the N9 error handling path in future integration.
8.  **Pre-routing Health Checks:** Future router execution should optionally run module health checks before routing.
9.  **Unavailable Route Handling:** If a required module is unavailable, route the execution to the N9 error handling path.
